# TASK06 - Rollout, Deprecation Policy And Safe Adoption For Existing Apps

## Цель

Подготовить безопасный rollout новых версий `nsg_data` и `nsg_controls` так, чтобы существующие приложения могли обновляться постепенно и предсказуемо.

Этот этап не про "дописать еще архитектуру", а про контролируемое внедрение.

## Основной вопрос этапа

Как выпустить новые пакеты так, чтобы:

- старые приложения продолжили работать
- новые приложения могли включать новые реализации
- команды приложений понимали, что обязательно, а что опционально

## Что нужно сделать

### 1. Определить deprecation policy

Нужно явно разделить:

- что остается supported legacy API
- что становится deprecated
- что нельзя удалять до следующего major cycle

### 2. Подготовить migration notes для существующих приложений

Должны быть отдельно описаны три сценария:

#### Сценарий A

Приложение просто обновляет `nsg_data` и `nsg_controls`, но продолжает жить на `GetX`.

#### Сценарий B

Приложение обновляет пакеты и начинает точечно использовать новые интерфейсы, оставаясь в основном на `GetX`.

#### Сценарий C

Приложение обновляет пакеты и начинает пилотные экраны/flow на `Riverpod`.

### 3. Подготовить release checklist

Нужно описать, что проверять перед выпуском:

- backward compatibility
- analyzer
- representative runtime flows
- demo/example
- migration docs completeness

### 4. Подготовить финальный roadmap after rollout

Нужно явно указать, что идет следующей волной:

- app-level migration
- cleanup
- router replacement, если он вообще понадобится позже

## Что нужно документировать

Нужно создать:

- `deprecation_policy.md`
- `adoption_scenarios.md`
- `release_checklist.md`
- `post_rollout_roadmap.md`

## Acceptance Criteria

- Есть понятная стратегия выпуска новых версий пакетов.
- Для существующих приложений описан путь обновления с минимальными изменениями.
- Deprecated vs supported API clearly documented.
- Новая архитектура готова не только технически, но и организационно к внедрению.

## Особое ограничение

Нельзя объявлять deprecated то, что существующие приложения реально обязаны использовать, если при этом для них еще нет столь же простого replacement path.
