# GetX Decoupling Program

## Зачем этот пакет документов

Этот набор задач описывает не миграцию конкретного приложения, а план перепроектирования `nsg_data` и `nsg_controls` так, чтобы:

- убрать прямую архитектурную зависимость базовых пакетов от `GetX`
- не трогать пока навигацию приложений как обязательную часть первого этапа
- сохранить работоспособность существующих приложений при обновлении пакетов
- дать возможность поверх одних и тех же базовых контрактов использовать и `GetX`, и `Riverpod`

## Главная цель

Перевести `GetX` из статуса "жестко вшитая основа пакетов" в статус "одна из реализаций интерфейсов".

Это значит:

- `nsg_data` не должен знать напрямую о `Get.find`, `GetStatus`, `Get.back`, `Get.dialog`, `Get.currentRoute`
- `nsg_controls` не должен зависеть напрямую от `Get.context`, `Get.width`, `Get.locale`, `Get.dialog`, `Get.snackbar`
- текущий `GetX`-мир должен продолжить работать через default adapter layer

## Ключевой принцип

### Минимум изменений в существующих приложениях

Это главный критерий успеха.

Обновление существующих приложений на новые версии `nsg_data` и `nsg_controls` должно требовать:

- либо ноль изменений
- либо очень небольшие и локальные изменения bootstrap-уровня

Не допускается дизайн, при котором после обновления пакетов каждое приложение обязано:

- срочно переписывать все контроллеры
- менять весь роутинг
- заменять `GetMaterialApp`
- переделывать все `Get.put`
- переписывать все экраны на новый state management

## Стратегия совместимости

Все задачи ниже должны выполняться с опорой на такую схему:

1. Сначала вводятся новые framework-agnostic интерфейсы.
2. Затем добавляются default `GetX` implementations этих интерфейсов.
3. Старый публичный API сохраняется, пока это возможно.
4. Старые классы получают внутреннюю делегацию на новые интерфейсы, а не удаляются сразу.
5. `Riverpod` support добавляется как вторая реализация, а не как замена в лоб.
6. Лишь после стабилизации можно начинать депрекейтить старые прямые зависимости.

## Что пока не делаем

На первом большом цикле этой программы не считаются обязательными:

- замена роутера приложений
- отказ от `GetMaterialApp`
- переписывание всех `Bindings`
- массовая миграция экранов приложений
- полный отказ от legacy GetX controller API

Все это может последовать потом, но не должно быть prerequisite для выпуска совместимой новой версии пакетов.

## Как использовать этот пакет

Задачи выполняются по порядку:

- `TASK01.md`
- `TASK02.md`
- `TASK03.md`
- `TASK04.md`
- `TASK05.md`
- `TASK06.md`

Каждый `TASK*.md` должен быть отдельным рабочим этапом, отдельной веткой или отдельным PR.

Нельзя перескакивать к поздним этапам, пока не выполнены ранние совместимые seam-этапы.

## Ожидаемый результат всей программы

После выполнения всех задач должно быть возможно:

- продолжать использовать старые `GetX`-приложения почти без изменений
- писать новые приложения или новые flow на `Riverpod`
- использовать `nsg_controls` без прямой привязки к `GetX`
- постепенно убирать legacy-зависимости уже на уровне приложений, а не пакетов
